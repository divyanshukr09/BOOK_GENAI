### 1.3.2 Virtual Environments

**What is a Virtual Environment?**
A virtual environment is an isolated, self-contained Python installation
that exists only for a single project. Think of it like a separate
lunchbox for each school subject — what you put in your Maths lunchbox
stays there and never mixes with what is in your Science lunchbox. In
Python, each project might need different versions of the same library.
Without virtual environments, all your projects share one global Python
installation, and installing a new library version for one project can
break another project entirely.

**Why Do You Need This?**
In professional AI development, you will simultaneously work on multiple
projects. Your RAG project might need LangChain 0.2 while your older
chatbot project needs LangChain 0.1. Without isolation, upgrading one
breaks the other. Virtual environments solve this completely.

**Real-World Scenario**
Imagine you are a data scientist at Swiggy working on two systems: an
older recommendation engine that uses scikit-learn 1.2, and a new GenAI
chatbot being built with the latest libraries. If both share the same
Python installation, upgrading scikit-learn for the new project would
break the recommendation engine's compatibility — potentially crashing
the production system that millions of users depend on. Virtual
environments give each project its own isolated sandbox.
